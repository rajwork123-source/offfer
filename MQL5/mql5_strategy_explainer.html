EMA Crossover Strategy
The EMA (Exponential Moving Average) crossover is one of the most fundamental and widely-used trading strategies. It works by tracking two moving averages — one fast-reacting, one slow-reacting — and trading whenever they cross. When the fast line crosses above the slow line, the market is gaining upward momentum. When it crosses below, downward momentum is building. The "exponential" part means recent prices are weighted more heavily than older ones, making it more responsive than a simple average.
Entry Signal
Fast EMA crosses above Slow EMA → Buy
Exit Signal
Fast EMA crosses below Slow EMA → Close
Default Periods
Fast: 9 | Slow: 21
Best Market
Trending markets (avoid ranging)

#property strict
Enables strict compiler mode. This forces MQL5 to catch more potential bugs at compile time — like using undeclared variables or mismatched types. It's like turning on safety warnings. Always include it. Why: Catches mistakes before they cost you money in live trading.
input int FastPeriod = 9
The input keyword creates a user-editable parameter. When you attach this EA to a chart, MT5 shows a dialog where you can change FastPeriod from 9 to anything else — no code changes needed. The default value is 9 (meaning the EMA will average the last 9 candles). Why 9 and 21? These are classic values — 9 reacts quickly to price moves, 21 moves more smoothly. You can experiment with other pairs like 5/20 or 10/50.
input double LotSize = 0.1
How many lots to trade per order. 0.1 lots = 1 mini lot = $1 per pip on most forex pairs. Keeping it as an input means the user can adjust risk without touching the code. Why not hardcode it? Flexibility — different account sizes need different lot sizes.

Global Variables & OnInit()
int emaFastHandle, emaSlowHandle; int OnInit() { emaFastHandle = iMA(_Symbol, _Period, FastPeriod, 0, MODE_EMA, PRICE_CLOSE); emaSlowHandle = iMA(_Symbol, _Period, SlowPeriod, 0, MODE_EMA, PRICE_CLOSE); if(emaFastHandle == INVALID_HANDLE || emaSlowHandle == INVALID_HANDLE) { Print("Error creating indicator handles!"); return(INIT_FAILED); } return(INIT_SUCCEEDED); }
int emaFastHandle
A handle is like a remote control for an indicator. When you call iMA(), MQL5 doesn't give you the EMA values directly — it gives you an integer "handle" (a reference number). You use this handle later to request actual values. Think of it like a library card number — you show it to get the book. Why this design? Handles let MT5 calculate indicators efficiently in the background without recalculating everything each tick.
iMA(_Symbol, _Period, FastPeriod, 0, MODE_EMA, PRICE_CLOSE)
This is the indicator creation function. Let's break down every argument:

• _Symbol — use the chart's current pair (e.g. EURUSD)
• _Period — use the chart's current timeframe (e.g. H1)
• FastPeriod — how many bars to average (our input, default 9)
• 0 — shift: 0 means no offset, indicator aligns with current bar
• MODE_EMA — use Exponential Moving Average formula (not simple/weighted)
• PRICE_CLOSE — calculate based on candle closing prices
Why PRICE_CLOSE? Close prices are the most important — they represent where buyers and sellers agreed to end the battle for that candle.
if handle == INVALID_HANDLE
Always check if indicator creation succeeded. If MT5 can't create the indicator (e.g. not enough history, wrong symbol), it returns INVALID_HANDLE (-1). Returning INIT_FAILED tells MT5 the EA failed to start cleanly, and MT5 will remove it from the chart automatically. Why: Without this check, the EA might silently trade with wrong data and cause unexpected losses.

OnTick() — The Trading Logic
double fastNow[2], slowNow[2]; void OnTick() { if(CopyBuffer(emaFastHandle, 0, 0, 2, fastNow) < 2) return; if(CopyBuffer(emaSlowHandle, 0, 0, 2, slowNow) < 2) return; bool crossedUp = (fastNow[1] > slowNow[1]) && (fastNow[0] <= slowNow[0]); bool crossedDown = (fastNow[1] < slowNow[1]) && (fastNow[0] >= slowNow[0]); if(crossedUp && PositionsTotal() == 0) trade.Buy(LotSize, _Symbol); if(crossedDown && PositionsTotal() > 0) trade.PositionClose(_Symbol); }
double fastNow[2]
An array to store 2 EMA values. We need TWO values — the current bar's EMA and the previous bar's EMA. Why? To detect a crossover we must compare: "was fast above slow before? Is it below now?" That requires looking at two moments in time. Index [0] = previous completed bar. Index [1] = the bar before that. (CopyBuffer fills arrays right-to-left: newest data goes to index 0.)
CopyBuffer(handle, 0, 0, 2, array)
CopyBuffer retrieves actual indicator values from a handle. Arguments: handle, buffer number (0 = main line), start position (0 = most recent), count (2 values), destination array. It returns how many values it copied — if less than 2, data isn't ready yet, so we return (skip this tick). Why check the return value? On the very first ticks when the EA starts, there may not be enough history to compute the EMA. Skipping those ticks prevents errors.
bool crossedUp = (fastNow[1] > slowNow[1]) && (fastNow[0] <= slowNow[0])
This is the crossover detection logic — the heart of the strategy. Let's decode it:

• fastNow[1] > slowNow[1] — on the PREVIOUS bar, fast EMA was ABOVE slow EMA
• fastNow[0] <= slowNow[0] — on the bar BEFORE THAT, fast was BELOW or equal
• Both conditions together = the fast line just crossed downward = BEARISH signal

The crossedUp version is the mirror: fast was below, now it's above = BULLISH. Why compare two consecutive bars? A single comparison only tells you current position, not direction of movement. Comparing two bars tells you it just changed — that's the signal.
PositionsTotal() == 0
Check if we already have an open trade before buying. PositionsTotal() returns the number of currently open positions. We only buy if we have none open — this prevents the EA from opening trade after trade every tick while the crossover condition is still true. Without this check, a bullish crossover lasting many ticks would open hundreds of buy orders!
💡
Why EMA and not SMA? A Simple Moving Average (SMA) gives equal weight to all bars. An EMA gives more weight to recent bars. This means the EMA reacts faster to new price moves — crossover signals come earlier, which is both a strength (enter trend sooner) and a weakness (more false signals in choppy markets).
⚠️
The biggest weakness of crossover strategies is "whipsawing" — when price oscillates around the crossover point repeatedly, generating many false buy/sell signals in a row. Always add a filter (like checking the ADX indicator to confirm a trend exists) before trading this live.

Oscillator Beginner Friendly
RSI Overbought / Oversold Strategy
The RSI (Relative Strength Index) is a momentum oscillator that measures the speed and magnitude of recent price changes. It produces a value between 0 and 100. Above 70 means the asset is "overbought" — it has risen too far, too fast — and a pullback may come. Below 30 means "oversold" — it has fallen too fast — and a bounce may be due. This strategy bets on those reversals.
Buy Signal
RSI crosses above 30 (from oversold)
Sell Signal
RSI crosses above 70 (overbought)
RSI Period
14 bars (industry standard)
Best Market
Ranging / sideways markets
1
RSI Indicator Handle
int rsiHandle; double rsiValues[3]; int OnInit() { rsiHandle = iRSI(_Symbol, _Period, 14, PRICE_CLOSE); if(rsiHandle == INVALID_HANDLE) return(INIT_FAILED); return(INIT_SUCCEEDED); }
iRSI(_Symbol, _Period, 14, PRICE_CLOSE)
Creates the RSI indicator. The 14 is the lookback period — RSI compares average gains vs average losses over the last 14 candles. This period was chosen by RSI's creator Welles Wilder and remains the default. Shorter periods (7) make RSI more sensitive but noisier. Longer (21) makes it smoother but slower. Why 14? It was originally calibrated for weekly charts. On daily charts it works well because 14 days covers about 3 trading weeks.
double rsiValues[3]
We copy 3 RSI values to detect when RSI crosses the 30 or 70 line. With 3 values we can check: was RSI below 30 two bars ago, did it recover above 30 last bar? That gives a clean "just crossed" signal rather than just "RSI is above 30 right now." More values = more context = more reliable signal detection.
2
RSI Trading Logic
void OnTick() { if(CopyBuffer(rsiHandle, 0, 0, 3, rsiValues) < 3) return; double rsiCurrent = rsiValues[0]; double rsiPrevious = rsiValues[1]; // BUY: RSI just crossed back above 30 (oversold recovery) bool buySignal = (rsiPrevious < 30) && (rsiCurrent >= 30); // SELL: RSI just crossed above 70 (overbought — exit longs) bool sellSignal = (rsiPrevious < 70) && (rsiCurrent >= 70); if(buySignal && PositionsTotal() == 0) trade.Buy(LotSize, _Symbol); if(sellSignal && PositionsTotal() > 0) trade.PositionClose(_Symbol); }
rsiValues[0] vs rsiValues[1]
CopyBuffer fills arrays so index [0] = most recent bar, [1] = previous bar. So rsiValues[0] is the RSI value RIGHT NOW, and rsiValues[1] is what RSI was on the last completed candle. By comparing these two, we can tell if RSI just crossed a threshold this tick. This is the correct way to detect a "just crossed" event — not just checking if the value is currently above/below the level.
rsiPrevious < 30 && rsiCurrent >= 30
The oversold recovery signal — one of the cleanest patterns in technical analysis. This checks: "was RSI in oversold territory (below 30) on the last bar, AND has it now recovered back above 30?" That recovery is the actual buy signal — not just "RSI is low." RSI being low means it's oversold. RSI crossing back above 30 means the selling pressure is easing and buyers may be stepping in. We wait for the CROSS, not just the level. RSI can stay below 30 for many bars in a downtrend — waiting for the recovery reduces false entries.
💡
How RSI is actually calculated: RSI = 100 - (100 / (1 + RS)) where RS = Average Gain over N periods / Average Loss over N periods. If average gain is 0.5% and average loss is 0.25%, RS = 2, RSI = 100 - (100/3) = 66.7. You don't need to calculate this yourself — iRSI() does it for you.
⚠️
RSI strategies fail in strong trends. In a powerful downtrend, RSI can stay below 30 for weeks. "Oversold" doesn't mean it will go up — it means it HAS gone down a lot. Always check the bigger timeframe trend direction before fading an RSI extreme.
Mean Reversion Intermediate
Bollinger Bands Strategy
Bollinger Bands consist of three lines plotted around price: a middle SMA (simple moving average), and two outer bands placed 2 standard deviations above and below it. The key insight is that about 95% of all price action falls within these bands. When price touches or pierces the outer band, it is statistically unusual — and a reversion back toward the middle band is likely.
Buy Signal
Price touches lower band → Buy
Exit Signal
Price reaches middle band (SMA)
Parameters
Period: 20 | Deviation: 2.0
Best Market
Ranging / low-volatility markets
1
Bollinger Band Handle & Buffers
int bbHandle; double bbUpper[2], bbMiddle[2], bbLower[2]; input int BBPeriod = 20; input double BBDeviation = 2.0; int OnInit() { bbHandle = iBands(_Symbol, _Period, BBPeriod, 0, BBDeviation, PRICE_CLOSE); if(bbHandle == INVALID_HANDLE) return(INIT_FAILED); return(INIT_SUCCEEDED); }
iBands(..., BBPeriod, 0, BBDeviation, ...)
iBands() creates Bollinger Bands. Key arguments:

• BBPeriod = 20 — the middle band is a 20-bar SMA. 20 is the standard (roughly one trading month)
• 0 — band shift (0 = no offset; 1 would shift the bands one bar forward)
• BBDeviation = 2.0 — the bands are placed 2 standard deviations from the middle. Statistically, 95.4% of a normal distribution falls within ±2 std devs
Why 2.0 deviations? It's a balance — 1.0 would be too tight (price touches bands constantly), 3.0 would be too loose (signals are too rare). 2.0 catches extremes without excessive signals.
bbUpper, bbMiddle, bbLower arrays
Bollinger Bands have 3 separate data buffers, each with their own index. When you call CopyBuffer, you need to specify which band you want: buffer 0 = middle band (SMA), buffer 1 = upper band, buffer 2 = lower band. That's why we need three separate arrays. Each buffer is like a separate data stream — middle shows trend, upper/lower show volatility extremes.
2
Band Touch Logic
void OnTick() { if(CopyBuffer(bbHandle, 0, 0, 2, bbMiddle) < 2) return; if(CopyBuffer(bbHandle, 1, 0, 2, bbUpper) < 2) return; if(CopyBuffer(bbHandle, 2, 0, 2, bbLower) < 2) return; double ask = SymbolInfoDouble(_Symbol, SYMBOL_ASK); double bid = SymbolInfoDouble(_Symbol, SYMBOL_BID); double close1 = iClose(_Symbol, _Period, 1); // last completed bar // Price touched lower band (oversold extreme) bool touchedLower = (close1 <= bbLower[1]); // Price is back at or above middle band (mean reversion target hit) bool reachedMiddle = (bid >= bbMiddle[0]); if(touchedLower && PositionsTotal() == 0) trade.Buy(LotSize, _Symbol, ask, ask - (bbUpper[0] - bbLower[0]) * 0.5, // SL: half band width bbMiddle[0]); // TP: at middle band if(reachedMiddle && PositionsTotal() > 0) trade.PositionClose(_Symbol); }
CopyBuffer(bbHandle, 1, ...) vs (bbHandle, 2, ...)
The second argument is the buffer index. For iBands specifically: 0 = middle SMA, 1 = upper band, 2 = lower band. Every indicator has its own buffer layout — you must check the MQL5 documentation for each one. Getting the buffer number wrong means you're reading the wrong data silently. Always double-check buffer indices in the MQL5 docs for each indicator you use.
close1 <= bbLower[1]
We use the LAST COMPLETED BAR's close price (bar index 1), not the current tick. The current bar (index 0) is still forming — its close price changes every tick. Using it would generate signals mid-candle that might not exist by candle close. Using the completed bar's close (index 1) gives a confirmed, stable signal. Trading on completed candles is more reliable — the signal won't disappear mid-bar like it might with bar[0].
ask - (bbUpper[0] - bbLower[0]) * 0.5
This calculates a dynamic stop loss based on band width. bbUpper - bbLower = total band width. Multiplying by 0.5 gives half the band width. This is subtracted from the ask price to place the stop loss below entry. The logic: if the bands are wide (high volatility), give the trade more room. If bands are narrow, less room needed. Why dynamic SL? A fixed-pip stop loss ignores current market conditions. Volatility-adjusted stops are more intelligent.
TP: bbMiddle[0]
The take profit is set exactly at the middle band (the 20-period SMA). This embodies the mean reversion concept: we entered because price went to an extreme, and we expect it to return to the "mean" (average). The middle band IS that mean. This gives the trade a clear mathematical target. The middle band is also a common area where price finds support/resistance, making it a natural exit zone.
📐
What "standard deviation" actually means in plain English: If most prices over 20 bars closed around 1.1000, then the "standard deviation" measures how spread out those prices were. A deviation of 0.0050 means most prices were within 0.0050 of the average. Multiplied by 2, the bands sit at 1.0900 and 1.1100. When price hits 1.0900, it's 2 standard deviations below average — a statistically rare extreme.
Mean Reversion Advanced
Mean Reversion Z-Score Strategy
This is a pure statistical mean reversion strategy. Instead of using indicator lines, it directly computes a Z-Score — a number that measures how many standard deviations the current price is from its recent average. A Z-Score of +2 means price is 2 standard deviations above average. A Z-Score of -2 means 2 below. These extremes suggest the price will likely "revert to the mean" — snap back toward the average.
Buy Signal
Z-Score < -2.0 (price far below mean)
Sell/Exit Signal
Z-Score > +2.0 (price far above mean)
Lookback Period
20 bars (configurable)
Target
Z-Score returns to 0 (the mean)
1
Computing Mean & Standard Deviation Manually
input int LookbackPeriod = 20; input double ZScoreThreshold = 2.0; double CalculateMean(int period) { double sum = 0; for(int i = 1; i <= period; i++) sum += iClose(_Symbol, _Period, i); return(sum / period); } double CalculateStdDev(double mean, int period) { double variance = 0; for(int i = 1; i <= period; i++) { double diff = iClose(_Symbol, _Period, i) - mean; variance += diff * diff; } return(MathSqrt(variance / period)); }
CalculateMean: sum / period
This manually computes the arithmetic mean (average) of the last N closing prices. The loop runs from bar 1 to bar LookbackPeriod (skipping bar 0 which is still forming). It accumulates all close prices into sum, then divides by the period. We start from bar 1, not bar 0, to only use confirmed closed candles — bar 0 is still live and changes every tick.
variance += diff * diff
This is the variance formula — the core of standard deviation. For each bar, we calculate how far its close is from the mean (diff). Then we square it (diff * diff). Squaring serves two purposes: it makes negative differences positive, and it amplifies large deviations more than small ones. We sum all these squared differences, then divide by period. Why square? You can't average positive and negative differences — they'd cancel out. Squaring makes everything positive so the average represents total spread.
MathSqrt(variance / period)
Standard deviation = square root of variance. Taking the square root "undoes" the squaring we did earlier, bringing the unit back to the same scale as the original prices. If variance is 0.000025, std dev = 0.005, meaning prices typically varied by ±0.005 from the mean. MathSqrt() is MQL5's built-in square root function. The result is in the same units as price, making it interpretable.
2
Z-Score Calculation & Trading
void OnTick() { double mean = CalculateMean(LookbackPeriod); double stddev = CalculateStdDev(mean, LookbackPeriod); double currentPrice = iClose(_Symbol, _Period, 1); if(stddev == 0) return; // Avoid division by zero double zScore = (currentPrice - mean) / stddev; Comment("Z-Score: ", DoubleToString(zScore, 2), " | Mean: ", DoubleToString(mean, 5)); if(zScore < -ZScoreThreshold && PositionsTotal() == 0) trade.Buy(LotSize, _Symbol); // Price far below mean — buy if(zScore > ZScoreThreshold && PositionsTotal() > 0) trade.PositionClose(_Symbol); // Price far above mean — exit }
zScore = (currentPrice - mean) / stddev
This is the Z-Score formula — one of the most important formulas in statistics. It answers: "How many standard deviations away from the mean is the current price?"

• Z = 0 means price is exactly at the mean (normal)
• Z = +2 means price is 2 standard deviations ABOVE mean (extreme high)
• Z = -2 means price is 2 standard deviations BELOW mean (extreme low)

Why divide by stddev? It normalizes the distance. Without it, a 50-pip move on USDJPY (which trades at 150) is very different from a 50-pip move on EURUSD (at 1.10). Z-Score makes them comparable.
if(stddev == 0) return
Critical safety check — prevents a division by zero crash. stddev would be 0 if all bars in the lookback period had exactly the same close price (impossible in real markets, but possible during weekend gaps or synthetic data). Division by zero in MQL5 doesn't crash the program but produces NaN (Not a Number), which will corrupt all calculations using that value downstream. Always validate before dividing. This is defensive programming — never assume your inputs are safe.
Comment("Z-Score: ", ...)
Comment() displays text on the chart in the top-left corner. This is invaluable for debugging and monitoring — you can see the Z-Score updating in real time as the EA runs. DoubleToString(value, 2) converts a double to a string with 2 decimal places. Always add visual feedback to your EAs during development. Watching the Z-Score move helps you understand how the strategy behaves before committing real money.
📊
Z-Score vs Bollinger Bands — what's the difference? They're mathematically almost identical! Bollinger Bands ARE a visual representation of Z-Score bands. When price touches the upper Bollinger Band, its Z-Score is approximately +2. The Z-Score strategy gives you the raw number directly, letting you set precise thresholds like 1.5 or 2.5, rather than being tied to a specific band width setting.
Trend Following Intermediate
MACD Strategy
MACD (Moving Average Convergence Divergence) is built from three components: the MACD line (difference between 12-period and 26-period EMAs), the Signal line (9-period EMA of MACD), and the Histogram (MACD minus Signal). When the MACD line crosses above the Signal line, it signals that short-term momentum is accelerating above long-term momentum — a bullish sign. It's a more sophisticated version of the EMA crossover that also quantifies how strong the trend is.
Buy Signal
MACD crosses above Signal line
Sell Signal
MACD crosses below Signal line
Parameters
Fast: 12 | Slow: 26 | Signal: 9
Extra Filter
Only trade when Histogram expands
1
MACD Handle & Three Buffers
int macdHandle; double macdLine[3], signalLine[3], histogram[3]; int OnInit() { macdHandle = iMACD(_Symbol, _Period, 12, 26, 9, PRICE_CLOSE); if(macdHandle == INVALID_HANDLE) return(INIT_FAILED); return(INIT_SUCCEEDED); }
iMACD(_Symbol, _Period, 12, 26, 9, PRICE_CLOSE)
Creates the MACD indicator with the classic parameters.

• 12 — Fast EMA period. Reacts quickly to recent price changes.
• 26 — Slow EMA period. Moves more smoothly, representing longer-term trend.
• 9 — Signal line period. A 9-period EMA of the MACD line itself, smoothing out noise.

MACD Line = EMA(12) - EMA(26). When this is positive, the fast EMA is above slow = bullish. When negative = bearish. The 12/26/9 combination was chosen by MACD's creator Gerald Appel and has remained the standard for 40+ years because it balances sensitivity and reliability.
macdLine, signalLine, histogram
MACD has 3 buffers: buffer 0 = MACD line, buffer 1 = Signal line, buffer 2 = Histogram. We create three separate arrays because we need all three for the full strategy. The histogram (MACD - Signal) is especially useful because when it's growing (increasing), momentum is accelerating in that direction. Using all three buffers gives much richer signals than just the crossover alone.
2
MACD Crossover with Histogram Confirmation
void OnTick() { if(CopyBuffer(macdHandle, 0, 0, 3, macdLine) < 3) return; if(CopyBuffer(macdHandle, 1, 0, 3, signalLine) < 3) return; if(CopyBuffer(macdHandle, 2, 0, 3, histogram) < 3) return; // Crossover: MACD crossed above Signal on previous bar bool bullCross = (macdLine[2] < signalLine[2]) && (macdLine[1] > signalLine[1]); // Histogram confirmation: momentum is growing (not fading) bool histGrowing = (histogram[1] > histogram[2]); bool bearCross = (macdLine[2] > signalLine[2]) && (macdLine[1] < signalLine[1]); if(bullCross && histGrowing && PositionsTotal() == 0) trade.Buy(LotSize, _Symbol); if(bearCross && PositionsTotal() > 0) trade.PositionClose(_Symbol); }
macdLine[2] vs macdLine[1]
We're looking at bars [2] and [1], not [0] and [1]. This is because we want to detect a crossover that happened on the LAST COMPLETED bar (index 1). To know it crossed, we need to compare it to the bar before that (index 2). Bar [0] is the current forming candle — we intentionally avoid it for confirmed signals. Waiting for bar close confirmation prevents reacting to intrabar MACD fluctuations that reverse before the candle closes.
histGrowing = histogram[1] > histogram[2]
This is an additional confirmation filter — one of the best enhancements to the basic MACD crossover. The histogram represents how fast the MACD and Signal lines are diverging. If the histogram is growing (getting more positive) after a bullish cross, it means momentum is accelerating in the buy direction. If histogram is already shrinking right after the cross, the move may already be fading. A crossover with growing histogram = strong signal. A crossover with shrinking histogram = weak signal, likely to fail. This single filter can significantly improve win rate.
💡
MACD vs EMA Crossover — why is MACD "better"? A 12/26 EMA crossover generates a signal when the two EMAs cross. MACD generates a signal when the DIFFERENCE between those EMAs crosses its own average. This second layer of averaging makes MACD signals slightly smoother and more reliable, with fewer false crossovers in choppy conditions. The histogram also gives you a preview of weakening momentum before the signal line actually crosses.
Breakout Intermediate
Donchian Channel Breakout Strategy
Donchian Channels plot the highest high and lowest low over N periods. When price breaks above the upper channel line, it means price has hit a new N-period high — this signals strong bullish momentum (breakout). The strategy made famous by the "Turtle Traders" experiment in the 1980s, which turned a group of non-traders into multi-millionaires by systematically following breakouts. The underlying idea: if price is making new highs, the trend is likely continuing.
Buy Signal
Price breaks above 20-bar high
Exit Signal
Price breaks below 10-bar low
Entry Channel
20 periods (breakout confirmation)
Exit Channel
10 periods (shorter = tighter exit)
1
Computing Donchian Channels Manually
input int EntryPeriod = 20; input int ExitPeriod = 10; double GetHighestHigh(int period) { double highest = iHigh(_Symbol, _Period, 1); for(int i = 2; i <= period; i++) { double h = iHigh(_Symbol, _Period, i); if(h > highest) highest = h; } return(highest); } double GetLowestLow(int period) { double lowest = iLow(_Symbol, _Period, 1); for(int i = 2; i <= period; i++) { double l = iLow(_Symbol, _Period, i); if(l < lowest) lowest = l; } return(lowest); }
double highest = iHigh(..., 1)
Initialize the highest value with bar 1's high, then compare against all other bars. Starting at bar 1 (not 0) means we only use closed candles. The loop then checks each subsequent bar — if any bar's high is higher than what we have, it becomes the new "highest." This is a simple linear scan for the maximum. Why not use MQL5's iHighest() built-in? We could! iHighest() does exactly this. Writing it manually helps you understand what's happening under the hood.
EntryPeriod=20, ExitPeriod=10
The asymmetry between entry and exit periods is intentional and important. A 20-bar breakout means price must make a new 4-week high to trigger entry — a meaningful level. The 10-bar exit (2-week low) triggers faster, giving back less profit when the trend reverses. This asymmetry was the core of the Turtle Traders' risk management: enter on big breakouts, exit on smaller reversals. Exit period < Entry period = you exit before another entry signal would form in the opposite direction. Keeps trades from overstaying their welcome.

Breakout Entry & ATR-Based Stop Loss
int atrHandle; int OnInit() { atrHandle = iATR(_Symbol, _Period, 14); if(atrHandle == INVALID_HANDLE) return(INIT_FAILED); return(INIT_SUCCEEDED); } void OnTick() { double atrValues[2]; if(CopyBuffer(atrHandle, 0, 0, 2, atrValues) < 2) return; double atr = atrValues[1]; // ATR of last completed bar double highest = GetHighestHigh(EntryPeriod); double lowest = GetLowestLow(ExitPeriod); double ask = SymbolInfoDouble(_Symbol, SYMBOL_ASK); double bid = SymbolInfoDouble(_Symbol, SYMBOL_BID); bool newHigh = (ask > highest); bool newLow = (bid < lowest); if(newHigh && PositionsTotal() == 0) trade.Buy(LotSize, _Symbol, ask, ask - (2 * atr), // Stop loss: 2 ATR below entry ask + (4 * atr)); // Take profit: 4 ATR above entry (2:1 RR) if(newLow && PositionsTotal() > 0) trade.PositionClose(_Symbol); }
iATR(_Symbol, _Period, 14)
ATR = Average True Range — the most important volatility measurement in trading. It calculates the average "range" of each candle over the last 14 bars, where range = max of: (High-Low), (|High-PrevClose|), (|Low-PrevClose|). A high ATR means the market is moving a lot. A low ATR means it's quiet. We use ATR to set our stop loss because fixed-pip stops are dumb — in a volatile market, a 20-pip stop will be hit by noise; in a quiet market, a 20-pip stop is too wide. ATR-based stops adapt to current conditions.
StopLoss: ask - (2 * atr)
The stop loss is placed 2 ATRs below the entry price. If ATR = 0.0050 (50 pips), then stop = entry - 0.0100 (100 pips below). This gives the trade enough room to breathe — normal market noise shouldn't exceed 2 ATRs. The Turtle Traders used exactly this: 2N (N = ATR) as their stop. Why 2x? 1x ATR is often too tight (hit by normal fluctuations). 3x ATR means you accept large losses. 2x is the sweet spot between giving the trade room and limiting damage.
TakeProfit: ask + (4 * atr)
The take profit is 4 ATRs above entry, giving a 2:1 reward-to-risk ratio. SL = 2 ATR, TP = 4 ATR means for every dollar risked, we target 2 dollars profit. With a 2:1 ratio, you only need to be right 34% of the time to break even. Breakout strategies are right maybe 40-50% of the time, which becomes profitable with this ratio. Risk:Reward ratio is arguably more important than win rate. A 30% win rate with 3:1 RR is more profitable than a 60% win rate with 0.5:1 RR. Always define your RR before trading.

⚠️
Breakout strategies suffer during ranging markets. If price oscillates between the same high and low repeatedly, you'll get false breakout signals above the high that quickly reverse. Always check if a market is trending (using ADX > 25) before applying a breakout strategy.
