The quality of your Pine Script output depends almost entirely on how you frame your request. A vague prompt gives vague code. A precise prompt gives production-ready code.
The anatomy of a great Pine Script prompt
A perfect prompt has four layers: platform context (Pine Script v5, TradingView), strategy type (indicator vs. strategy, script purpose), logic specification (exact entry/exit conditions), and output requirements (what to plot, alerts, parameter inputs).
❌ Weak Prompt
make me an rsi strategy
✅ Strong Prompt
Write a Pine Script v5 strategy() that enters LONG when RSI(14) crosses above 30 from below, and enters SHORT when RSI crosses below 70 from above. Use 1% of equity per trade. Include configurable RSI length as input. Plot RSI with overbought/oversold lines. Add alerts for each entry.
Prompt templates that work every time
🎯 First Script
Write a Pine Script v5 strategy() that trades [YOUR LOGIC]. Use strategy.percent_of_equity for position sizing at 1%. Make key values into input() parameters. Plot signals on chart.
Always specify v5, strategy vs indicator, and ask for inputs — saves 80% of follow-up requests.
🔧 Fixing Errors

Always paste the full error AND the full code. Never paraphrase errors — exact messages matter.
📊 Adding Indicators

One feature at a time. Adding multiple things in one prompt causes logic conflicts.

Alerts are a Pine Script superpower. Always add them so you can fire webhooks to your phone or bot.
⚠️
The v4 vs v5 Trap
Always specify "Pine Script v5" in your prompt. If you don't, Claude might generate v4 code which uses deprecated syntax like study() instead of indicator(), or the old security() patterns. TradingView still runs v4 but warns you, and some features only exist in v5.
Chapter 03 — First Strategy
Build your

strategy from scratch
A complete walkthrough: from an idea in plain English to a working TradingView strategy with entries, exits, stop losses, and a backtest. No prior Pine Script knowledge needed.
The strategy we'll build
A classic EMA Crossover strategy — long when the 9 EMA crosses above the 21 EMA, short when it crosses below. Simple, widely understood, and a perfect first script to understand the Pine Script structure before making it more complex.
Write a Pine Script v5 strategy() called "EMA Cross Strategy". Rules: - Enter LONG when ema(9) crosses ABOVE ema(21) - Enter SHORT when ema(9) crosses BELOW ema(21) - Close opposite position before entering new one - Use strategy.percent_of_equity = 2% per trade - Stop loss: 1.5x ATR(14) from entry price - Take profit: 3x ATR(14) from entry price (2:1 R:R) Inputs: fast EMA length, slow EMA length, ATR length, ATR multiplier for SL, ATR multiplier for TP Plot: fast EMA (blue), slow EMA (orange) on main chart Plot: background color green when in long, red when in short Add alertcondition() for long entry and short entry
Understanding the generated code
Claude will generate a complete script. Here's the structure you'll see — and what each section does:
ema_cross_strategy.pine
// @version=5 ← Always v5. Required first line. strategy("EMA Cross Strategy", overlay=true, default_qty_type=strategy.percent_of_equity, default_qty_value=2) // ── Inputs ───────────────────────────────────────── fastLen = input.int(9, "Fast EMA", minval=1) slowLen = input.int(21, "Slow EMA", minval=1) atrLen = input.int(14, "ATR Length") slMult = input.float(1.5, "SL ATR Mult") tpMult = input.float(3.0, "TP ATR Mult") // ── Indicators ────────────────────────────────────── fastEMA = ta.ema(close, fastLen) slowEMA = ta.ema(close, slowLen) atr = ta.atr(atrLen) // ── Signals ───────────────────────────────────────── longSignal = ta.crossover(fastEMA, slowEMA) shortSignal = ta.crossunder(fastEMA, slowEMA) // ── Entries ───────────────────────────────────────── if longSignal strategy.entry("Long", strategy.long) strategy.exit("Long SL/TP", "Long", stop=close - atr * slMult, limit=close + atr * tpMult) if shortSignal strategy.entry("Short", strategy.short) strategy.exit("Short SL/TP", "Short", stop=close + atr * slMult, limit=close - atr * tpMult) // ── Plots ─────────────────────────────────────────── plot(fastEMA, "Fast EMA", color=color.blue, linewidth=2) plot(slowEMA, "Slow EMA", color=color.orange, linewidth=2) bgcolor(strategy.position_size > 0 ? color.new(color.green, 90) : strategy.position_size < 0 ? color.new(color.red, 90) : na) // ── Alerts ────────────────────────────────────────── alertcondition(longSignal, "Long Entry", "EMA Cross: LONG signal") alertcondition(shortSignal, "Short Entry", "EMA Cross: SHORT signal")
Adding it to TradingView
1
Open Pine Editor
On TradingView, click "Pine Editor" at the bottom of any chart. It's the code editor tab. Delete the default template.
2
Paste the code
Paste the Claude-generated code into the Pine Editor. Make sure the first line is exactly //@version=5.
3
Click "Add to chart"
The blue "Add to chart" button compiles the script and applies it to the current chart. Errors appear in a red bar at the bottom.
4
Open Strategy Tester
Click the "Strategy Tester" tab next to Pine Editor. You'll see all trades, net profit, drawdown, and win rate automatically calculated.
🔥
Pro Tip: overlay=true vs false
If your strategy plots on the main price chart (like EMAs), use strategy("Name", overlay=true). If you're plotting something in a separate pane (like RSI), use overlay=false. Always tell Claude which you want.
Chapter 04 — Testing
Running &
backtesting your
Pine Script strategy
TradingView's Strategy Tester is built-in and instant. But raw backtest numbers lie. Here's how to read them correctly and ask the right questions.
The Strategy Tester panel
When you add a strategy() to the chart, the Strategy Tester tab appears automatically. It shows every trade in sequence, the equity curve, and the performance summary. Click the "Performance Summary" tab to see the key metrics.
MetricWhat it meansTarget
Net Profit %Total return over the test periodPositive, ideally >buy & hold
Profit FactorGross profit ÷ gross loss. Above 1 = profitable>1.5 for a solid strategy
Max Drawdown %Largest peak-to-trough drop in equity<20% to stay emotionally tradeable
Win Rate %% of trades that were profitableContext-dependent — a 40% WR can beat 70%
Avg TradeAverage profit/loss per tradeMust be positive after commissions
Total TradesNumber of trades in the test period>100 for statistical significance
🚨
Lookahead Bias — The Silent Killer
The most common Pine Script backtest error is using security() or higher timeframe data incorrectly, causing the strategy to "see the future." Always tell Claude: "Make sure there is no lookahead bias — use barmerge.lookahead_off for all security() calls." If your backtest looks too good, this is usually why.
Adding realistic commission & slippage
Update my strategy() declaration to include: - commission_type = strategy.commission.percent - commission_value = 0.05 (0.05% per trade, like Binance) - slippage = 2 (2 ticks of slippage per trade) This simulates real trading costs. Here is the current strategy() line: [paste it]
⚠️
Backtesting ≠ Future Performance
A Pine Script backtest uses historical data that the strategy never traded in real time. The further back the data and the more robust across multiple symbols and timeframes, the more trustworthy the backtest. Always test on at least 2 years of data and multiple market conditions (trending + ranging periods).

Chapter 06 — Optimization
Tuning your
Pine Script
for real performance
TradingView has a built-in Strategy Optimizer. But optimization without discipline creates curve-fitted garbage. Here's how to do it right with AI assistance.
Using TradingView's Strategy Optimizer
TradingView Pro+ and above includes an Optimizer (the flask icon in Strategy Tester). It tests parameter combinations and ranks them by any metric you choose. Always optimize for Profit Factor, not Net Profit — it's more robust.
1
Make all parameters inputs
Ask Claude: "Convert all hardcoded values to input() parameters with min, max, and step values." Only input() parameters appear in the Optimizer.
2
Set reasonable ranges
Don't optimize EMA from 1 to 500 — you'll find noise. Ask Claude to suggest reasonable ranges based on the strategy type.
3
Optimize in-sample, validate out-of-sample
Use the Date Range feature in Strategy Tester to optimize on 2020–2022, then test the winner on 2023–2025 without touching it.
4
Test across symbols and timeframes
A robust strategy works on multiple instruments. Ask Claude to help you understand which parameters are universal vs. symbol-specific.
🚨
Overfitting — The Biggest Mistake in Pine Script
If you optimize 10 parameters on 2 years of Bitcoin data, you're not finding a strategy — you're memorising Bitcoin's 2022–2024 price action. The more parameters you optimize, the more likely you're overfitting. Rule: fewer than 5 optimized parameters, always validate on out-of-sample data, always test on at least 2 different symbols.
Chapter 07 — Alerts
Alerts &
automation
from Pine Script
Pine Script's alert system is one of TradingView's killer features. Fire webhooks to Telegram, Discord, your phone, or auto-trading bots — all from a few lines of Pine Script.
Three ways to trigger alerts
Pine Script supports two alert mechanisms: alertcondition() creates named alert templates you set up in TradingView's alert panel, and alert() fires dynamically from within conditions. Strategy scripts also auto-generate alerts on strategy.entry() and strategy.exit() calls.
alerts_example.pine
// Method 1: alertcondition() — creates named templates alertcondition(longSignal, "Long Entry", "{{ticker}} LONG @ {{close}} | EMA Cross") // Method 2: alert() — fires dynamically in conditions if longSignal alert("LONG: " + syminfo.ticker + " @ " + str.tostring(close), alert.freq_once_per_bar) // Method 3: Strategy auto-alerts (built-in) // When using strategy.entry(), TradingView automatically // creates "Order filled" alerts you can enable in the // alert dialog by selecting "Order fills" event type // Webhook JSON payload for auto-trading bots: alertcondition(longSignal, "Long JSON", '{"action":"buy","symbol":"{{ticker}}","price":{{close}}}')

Walk-forward testing prompts
The gold standard for strategy validation. Divide history into windows, optimize each window in-sample, test on the next out-of-sample window. Claude can help you interpret the results.
I ran a walk-forward test on my Pine Script strategy. Period 1 (Jan 2020–Dec 2021, in-sample): PF=1.68, Win%=52%, MaxDD=14% Period 2 (Jan 2022–Dec 2023, out-of-sample): PF=1.29, Win%=47%, MaxDD=21% Period 3 (Jan 2024–present, forward test): PF=1.12, Win%=44%, MaxDD=24% Analyze this degradation pattern. Is it normal? What does it suggest about robustness? What Pine Script changes would improve walk-forward stability without overfitting?

Chapter 09 — Ship It
Pre-live deployment
checklist
Before you enable real-money alerts from your Pine Script strategy — go through every item. This checklist was built from the mistakes TradingView traders make most often.
Code Quality
Script loads without errors or warnings on TradingView. The status bar below Pine Editor must show no red messages.
First line is //@version=5. Not v4. Not missing. Always explicitly set the version.
All key values are input() parameters, not hardcoded. This lets you adjust without editing code and enables the Optimizer.
No lookahead bias — all request.security() calls use barmerge.lookahead_off. Ask Claude to audit your code for this specifically.
You can explain every signal condition in plain English. If you can't, you don't understand what you're trading.
Strategy Validation
Backtested with realistic commission and slippage set in the strategy() declaration. A 0-commission backtest is fantasy.
Profit Factor above 1.4 across at least 3 years of data. Not just the best year — consistent across years.
Maximum drawdown under 25% in backtest. Live trading always has higher drawdown than backtests due to execution differences.
Tested on out-of-sample date range — a date range NOT used for optimization. Profit Factor degradation under 30% is acceptable.
More than 150 total trades in the backtest. Fewer trades = results could be statistical noise.
Alerts & Risk
All alertcondition() calls tested manually by watching the chart and confirming alerts fire at the correct moment.
Every trade has a stop loss defined in the strategy.exit() call. An uncapped loss is a blown account waiting to happen.
Position sizing set to % of equity, not a fixed dollar amount. Use strategy.percent_of_equity and risk no more than 2% per trade.
Paper traded for at least 2 weeks with live alerts (but no real money). Confirm signals match your expectations in real conditions.
