Chapter 1 · Language Structure
How Pine Script Is Built
Pine Script is a domain-specific language purpose-built for financial chart scripting. Understanding its execution model is the single most important concept — everything else flows from it.
⚡
The Bar-by-Bar Execution Model
The most critical concept in Pine Script

Pine Script does not run your code once. It runs your entire script once per bar, starting from the oldest historical bar and moving forward in time to the most recent bar. Every line of your code executes on every bar.

This is fundamentally different from languages like Python where you write loops yourself. In Pine, the loop is implicit — TradingView loops over all bars and calls your script on each one.

Pine Script v6 — execution model
//@version=6 indicator("Execution Demo") // This runs on EVERY bar, left to right // On bar 1: close = 100.0, bar_index = 0 // On bar 2: close = 102.5, bar_index = 1 // On bar 3: close = 99.0, bar_index = 2 // ...and so on for all 5000+ bars plot(close, "Close") // plots one point per bar
🕰️ Historical Bars
Script runs on every past bar in sequence. You can access past values with the [] operator: close[1] = previous bar's close.
🔴 Realtime Bar
The last bar (current candle) updates tick-by-tick. Your script re-executes on every new tick until the bar closes.
📌 Bar Index
bar_index is 0 on the first (oldest) bar and increases by 1 per bar. Useful for counting bars and time-based logic.
⚡ Performance
Scripts must run fast — TradingView limits computation per bar. Avoid heavy nested loops; use built-in functions which are optimized.
🏗️
Script Structure
The anatomy of every Pine file
Pine Script v6 — full structure
// 1. Version declaration — ALWAYS first line //@version=6 // 2. Script type declaration — second line // One of: indicator(), strategy(), library() indicator("My Indicator", overlay=true, max_bars_back=500) // 3. Inputs — user-configurable parameters length = input.int(14, "RSI Length", minval=1) source = input.source(close, "Source") // 4. Calculations — your logic rsiValue = ta.rsi(source, length) // 5. Conditions — signals isOverbought = rsiValue > 70 isOversold = rsiValue < 30 // 6. Output — plots, shapes, labels, alerts plot(rsiValue, "RSI", color.blue) hline(70, "Overbought", color.red) hline(30, "Oversold", color.green)
ℹ️
Order matters: In Pine Script, you cannot use a variable before you declare it. This is unlike some languages. Declare calculations before you use them in conditions, and conditions before you use them in plots.

Pine Script v6 — type declarations
//@version=6 indicator("Types Demo") // Explicit type declarations (optional but clear) int myLength = 14 float myLevel = 70.5 bool myFlag = true string myLabel = "RSI Signal" color myColor = color.blue // Type inference (Pine figures it out) period = 20 // inferred as int price = close * 1.0 // inferred as float signal = close > open // inferred as bool // na — not available (missing value) float emptyVal = na isNA = na(close[5000]) // true if bar doesn't exist

The na trap: Many beginners write if value == na which never evaluates to true in Pine Script. Always use na(value) to check for missing data, never the == operator with na.
🔑
Variable Declaration Keywords
var, varip, and plain assignment
Pine Script v6 — var vs varip
//@version=6 indicator("Variable Keywords") // Plain assignment — recalculated fresh on EVERY bar price = close * 1.05 // new value each bar, previous value lost // var — initialized ONCE (on first bar), then PERSISTS // Perfect for counters, running totals, state machines var int barCount = 0 var float runningHigh = na barCount := barCount + 1 runningHigh := math.max(nz(runningHigh, high), high) // varip — like var but updates intra-bar (on each tick) // Rarely needed; use only for realtime tick-sensitive logic varip int tickCount = 0 tickCount := tickCount + 1 plot(barCount, "Bar Count")
💡
The := operator: In Pine, = is for first declaration, and := is for reassignment of an existing variable. If you use var, you always reassign with :=. Confusing these is a very common beginner mistake.

Chapter 3 · Series & Bars
Understanding Series — Pine's Core Data Model
A "series" in Pine Script is a sequence of values — one value per bar. Almost everything in Pine is a series. Mastering how to access and reference series data is fundamental to writing correct scripts.
📈
What Is a Series?
The timeline of values across all bars

When you write close in Pine Script, you're not reading a single number — you're referencing the entire series of closing prices, one value per bar. On the current bar, close gives you the current bar's close. On the previous bar's execution, it was the previous close. Pine handles this automatically.

Pine Script v6 — series & the [] history operator
//@version=6 indicator("Series Demo") // close is a series — current bar's close currentClose = close // close[1] = previous bar's close (1 bar ago) prevClose = close[1] // close[2] = 2 bars ago twoBarsAgo = close[2] // Any calculated value is also a series sma20 = ta.sma(close, 20) prevSMA = sma20[1] // previous bar's SMA value // Crossover detection using series history crossedAbove = close > sma20 and close[1] <= sma20[1]
💡
ta.crossover() is cleaner: Instead of manually comparing close > sma20 and close[1] <= sma20[1], Pine provides ta.crossover(close, sma20) which does the same thing. Built-in cross functions handle edge cases better.

Chapter 5 · Control Flow
if, for, while, and switch
Control flow in Pine Script controls which code runs on each bar. Pine is indentation-sensitive — blocks are defined by consistent indentation (4 spaces recommended), not curly braces.
🔀
if / else if / else
Pine Script v6 — if statements
//@version=6 indicator("Control Flow") rsi = ta.rsi(close, 14) // Basic if/else if rsi > 70 label.new(bar_index, high, "OB", color=color.red) else if rsi < 30 label.new(bar_index, low, "OS", color=color.green) else label.new(bar_index, close, "N", color=color.gray) // if as expression (returns a value) trendDir = if close > ta.sma(close, 200) "Bullish" else "Bearish"
🔁
for Loops
Pine Script v6 — for loops
//@version=6 indicator("For Loop Demo") // Count bullish bars in last 10 periods bullCount = 0 for i = 0 to 9 if close[i] > open[i] bullCount := bullCount + 1 // Loop with step (count down) for i = 10 to 0 by -1 // do something going backward // break and continue work as expected for i = 0 to 100 if i == 50 break // exit loop if i % 2 == 0 continue // skip to next iteration plot(bullCount, "Bull Count")
⚠️
Avoid heavy loops: Pine Script has a computation limit per bar. Long loops on every bar will hit the limit and cause a runtime error. For most calculations, built-in functions like ta.sma(), ta.highest() etc. are far more efficient than manual loops.
🔄
while Loops
Pine Script v6 — while loop
// while loop — runs while condition is true // Use sparingly — same performance concerns as for loops i = 0 total = 0.0 while i < 10 total := total + close[i] i := i + 1 avg = total / 10
🔀
switch Statement
Pine Script v6 — switch
// switch — cleaner than long if/else chains maType = input.string("SMA", options=["SMA","EMA","WMA"]) maValue = switch maType "SMA" => ta.sma(close, 20) "EMA" => ta.ema(close, 20) "WMA" => ta.wma(close, 20) => ta.sma(close, 20) // default case plot(maValue)
Chapter 6 · Functions
Writing and Using Functions
Pine Script lets you define your own functions to encapsulate reusable logic. Functions can return single values, tuples, or work with series data.
🛠️
Defining Functions
Pine Script v6 — function syntax
//@version=6 indicator("Functions Demo") // Basic function — single return value percentChange(current, previous) => (current - previous) / previous * 100 // Call the function pctChange = percentChange(close, close[1]) // Function with local variables trueRange(high_, low_, close_) => r1 = high_ - low_ r2 = math.abs(high_ - close_[1]) r3 = math.abs(low_ - close_[1]) math.max(r1, math.max(r2, r3)) // last expression is return value tr = trueRange(high, low, close) // Tuple return — multiple values getBands(src, length, mult) => basis = ta.sma(src, length) dev = mult * ta.stdev(src, length) upper = basis + dev lower = basis - dev [basis, upper, lower] // returns three values // Destructure the tuple [mid, upper, lower] = getBands(close, 20, 2.0) plot(mid, "Mid", color.blue) plot(upper, "Upper", color.red) plot(lower, "Lower", color.green)
ℹ️
No explicit return keyword: In Pine Script, the last expression in a function body is automatically the return value. You don't write return. For multiple return values, use tuple syntax [a, b, c] on the last line.
📤
Method Syntax (Pine v6)
Object-oriented style calling
Pine Script v6 — methods
// Pine v6 supports method syntax on types // Instead of: ta.sma(close, 20) // You can write: close.sma(20) myMA = close.sma(20) // equivalent to ta.sma(close, 20) myEMA = close.ema(20) // equivalent to ta.ema(close, 20) myRSI = close.rsi(14) // equivalent to ta.rsi(close, 14) // Useful for chaining on arrays arr = array.new_float(0) arr.push(close) arr.push(open) avg = arr.avg()

Chapter 11 · Alerts
Creating Alerts — Push Notifications from Pine
Alerts let your Pine Script notify you via email, push notification, webhook, or SMS when a condition is met. They work independently of whether you're watching the chart.
🔔
alert() Function
Pine Script v6 — alerts
//@version=6 indicator("Alert Demo") rsi = ta.rsi(close, 14) crossedOversold = ta.crossover(rsi, 30) // Basic alert — fires once when condition first becomes true if crossedOversold alert("RSI crossed above 30 — potential buy signal", alert.freq_once_per_bar) // Alert with dynamic message if crossedOversold msg = "RSI Buy Signal on " + syminfo.ticker + " | RSI: " + str.tostring(rsi, "#.##") + " | Price: " + str.tostring(close) alert(msg, alert.freq_once_per_bar) // Alert frequency options: // alert.freq_all — every tick while condition is true // alert.freq_once_per_bar — once per bar (when bar opens with condition true) // alert.freq_once_per_bar_close — once per bar on bar CLOSE
📡
alertcondition() — Classic Alert Setup
Pine Script v6 — alertcondition
// alertcondition — defines an alert that users set up manually // This appears in the "Create Alert" dialog under "Condition" buyCondition = ta.crossover(ta.ema(close,9), ta.ema(close,21)) sellCondition = ta.crossunder(ta.ema(close,9), ta.ema(close,21)) alertcondition(buyCondition, title="EMA Buy", message="EMA 9 crossed above EMA 21 — Buy signal") alertcondition(sellCondition, title="EMA Sell", message="EMA 9 crossed below EMA 21 — Sell signal") // Webhook alerts — send JSON to a URL (for bots) // Set the webhook URL in the alert dialog, then use: webhookMsg = '{"action":"buy","symbol":"' + syminfo.ticker + '","price":' + str.tostring(close) + '}' if buyCondition alert(webhookMsg, alert.freq_once_per_bar_close)
💡
Webhook for automation: If you want to connect Pine Script alerts to a trading bot, use alert() with a JSON message body and configure a webhook URL in TradingView's alert dialog. The JSON will be POSTed to your server on every trigger.

Chapter 13 · Multi-Timeframe
Accessing Higher Timeframe Data
One of Pine Script's most powerful features is the ability to fetch data from a different timeframe or symbol than the current chart. The request.* namespace handles all cross-context data.
🔭
request.security()
Pine Script v6 — request.security
//@version=6 indicator("MTF Demo", overlay=true) // Fetch daily SMA even when on 1H chart dailySMA = request.security( symbol = syminfo.tickerid, // same symbol, different TF timeframe = "D", // daily timeframe expression = ta.sma(close, 50), // what to fetch lookahead = barmerge.lookahead_off // avoid future leak! ) // Fetch data from a different symbol spxClose = request.security("SP:SPX", timeframe.period, close) // Fetch OHLC tuple from higher timeframe [wOpen, wHigh, wLow, wClose] = request.security( syminfo.tickerid, "W", [open, high, low, close] ) plot(dailySMA, "Daily SMA 50", color.orange, 2)
⚠️
CRITICAL — avoid lookahead bias: Always use lookahead=barmerge.lookahead_off. The alternative lookahead_on allows future data to leak into past bars, making backtests look impossibly good. Real trading will fail. This is the most common MTF mistake.

Chapter 14 · Libraries
Pine Script Libraries — Reusable Code Modules
Libraries are Pine Script scripts whose functions can be imported and used by other scripts. They're the professional way to share and reuse code across multiple indicators and strategies.
🏛️
Creating a Library
Pine Script v6 — library definition
//@version=6 // @description Utility functions for trend analysis library("TrendUtils", overlay=true) // @function Detects if price is in a higher-high, higher-low uptrend // @param lookback Number of bars to analyze // @returns bool — true if uptrend structure confirmed export isUptrend(simple int lookback = 20) => highHigh = high > ta.highest(high, lookback)[1] lowHigh = low > ta.lowest(low, lookback)[1] highHigh and lowHigh // @function Calculate normalized ATR as a percentage of price export atrPercent(simple int len = 14) => ta.atr(len) / close * 100
📥
Importing a Library
Pine Script v6 — importing libraries
//@version=6 indicator("My Indicator Using Library") // Import your published library // Format: import AuthorUsername/LibraryName/version as alias import YourUsername/TrendUtils/1 as trend // Import a popular community library import PineCoders/DebugLib/1 as debug // Use the library functions via the alias uptrend = trend.isUptrend(20) atrPct = trend.atrPercent(14) bgcolor(uptrend ? color.new(color.green, 90) : na) plot(atrPct, "ATR %")
